1. Архитектурный подход REST API (CRUD), RESTful веб-сервисы

REST (Representational State Transfer) — это архитектурный стиль для построения распределенных систем, основанный на принципах веба и протокола HTTP. Разработан Роем Филдингом в 2000 году.

Основные принципы REST:
  • Stateless (без состояния): каждый запрос содержит всю необходимую информацию, сервер не хранит состояние клиента между запросами
  • Клиент-сервер: разделение ответственности — клиент отвечает за UI, сервер за данные
  • Кэширование: ответы могут быть помечены как кэшируемые для повышения производительности
  • Единообразие интерфейса: стандартизированные методы взаимодействия
  • Слоистая система: клиент не знает, с каким именно сервером взаимодействует (может быть прокси, балансировщик)
CRUD операции и HTTP методы:
  • Create (создание): POST /users — создать нового пользователя
  • Read (чтение): GET /users/123 — получить пользователя, GET /users — список всех
  • Update (обновление): PUT /users/123 — полное обновление, PATCH /users/123 — частичное
  • Delete (удаление): DELETE /users/123 — удалить пользователя
RESTful веб-сервисы:
  • Используют URI для идентификации ресурсов (существительные, не глаголы): /users, /products/42
  • Возвращают данные в стандартных форматах: JSON (чаще всего), XML
  • Используют стандартные коды ответов HTTP: 200 OK, 201 Created, 404 Not Found, 500 Internal Server Error
  • Поддерживают версионирование API: /api/v1/users, /api/v2/users
  • Могут использовать HATEOAS (Hypermedia As The Engine Of Application State) — ссылки на связанные ресурсы в ответах

Преимущества: простота, масштабируемость, независимость от платформы, кэширование, видимость (легко отладить через браузер).

2. Понятие протокол. Виды протоколов

Протокол — это формальный набор правил и соглашений, определяющих формат и порядок обмена сообщениями между двумя или более взаимодействующими объектами в сети, а также действия, предпринимаемые при передаче и приеме сообщений.

Протоколы определяют: синтаксис (формат данных), семантику (значение полей), тайминг (скорость передачи, последовательность).

Классификация по уровням модели OSI:
  • Прикладной уровень (7): HTTP/HTTPS (веб), FTP (передача файлов), SMTP (почта), DNS (имена), SSH (удаленный доступ), Telnet, SNMP (управление сетью)
  • Представления (6): SSL/TLS (шифрование), MIME (форматы данных)
  • Сеансовый (5): NetBIOS, RPC (удаленный вызов процедур)
  • Транспортный (4): TCP (надежная доставка), UDP (быстрая доставка), SCTP
  • Сетевой (3): IP (маршрутизация), ICMP (диагностика), ARP (преобразование адресов), OSPF, BGP (маршрутизация)
  • Канальный (2): Ethernet, Wi-Fi (802.11), PPP, HDLC
  • Физический (1): стандарты кабелей, радиосигналов (RS-232, USB)
По назначению:
  • Маршрутизации: RIP, OSPF, BGP — определяют пути передачи данных
  • Безопасности: IPsec, SSL/TLS, SSH — защита данных
  • Управления: SNMP, ICMP — мониторинг и диагностика сети
  • Конфигурации: DHCP, BOOTP — автоматическая настройка устройств

Стандартизация: протоколы разрабатываются организациями IETF (Internet Engineering Task Force), IEEE, ISO, ITU.

3. Протоколы HTTP/HTTPS

HTTP (HyperText Transfer Protocol) — протокол прикладного уровня для передачи гипертекстовых документов. Работает по модели запрос-ответ.

Характеристики HTTP:
  • Порт: 80 (по умолчанию)
  • Stateless: не сохраняет состояние между запросами (используются cookies, сессии для обхода)
  • Текстовый протокол: легко читается человеком
  • Версии: HTTP/1.0 (1996), HTTP/1.1 (1997, persistent connections), HTTP/2 (2015, мультиплексирование), HTTP/3 (2022, QUIC)
HTTP методы:
  • GET: получение ресурса (идемпотентный, безопасный)
  • POST: создание ресурса, отправка данных (не идемпотентный)
  • PUT: полная замена ресурса (идемпотентный)
  • PATCH: частичное обновление ресурса
  • DELETE: удаление ресурса (идемпотентный)
  • HEAD: получение заголовков без тела ответа
  • OPTIONS: получение поддерживаемых методов
Коды состояния HTTP:
  • 1xx (информационные): 100 Continue, 101 Switching Protocols
  • 2xx (успех): 200 OK, 201 Created, 204 No Content
  • 3xx (перенаправление): 301 Moved Permanently, 302 Found, 304 Not Modified
  • 4xx (ошибка клиента): 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found
  • 5xx (ошибка сервера): 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable
HTTPS (HTTP Secure):
  • Порт: 443
  • Шифрование: использует SSL/TLS для защиты данных
  • Процесс: TLS handshake → обмен ключами → шифрованная передача данных
  • Сертификаты: проверка подлинности сервера через Certificate Authority (CA)
  • Защита: конфиденциальность (шифрование), целостность (защита от изменений), аутентификация (проверка сервера)
  • Преимущества: защита от перехвата, MITM атак, улучшение SEO, доверие пользователей
4. SOAP, структура протокола

SOAP (Simple Object Access Protocol) — протокол обмена структурированными сообщениями в распределенных системах, основанный на XML. Разработан для обеспечения взаимодействия между приложениями независимо от платформы и языка программирования.

Структура SOAP сообщения:
  • Envelope (конверт): корневой элемент, обязательный. Определяет начало и конец сообщения, содержит пространства имен XML. Формат: <soap:Envelope>...</soap:Envelope>
  • Header (заголовок): необязательный элемент. Содержит метаинформацию: аутентификацию, транзакции, маршрутизацию, приоритеты. Атрибуты: mustUnderstand (обязательность обработки), actor (получатель)
  • Body (тело): обязательный элемент. Содержит основные данные запроса или ответа, вызовы методов и параметры
  • Fault (ошибка): необязательный элемент внутри Body. Содержит информацию об ошибках: faultcode (код ошибки), faultstring (описание), faultactor (источник), detail (детали)
Характеристики SOAP:
  • Транспорт: может работать поверх HTTP, SMTP, TCP, JMS
  • Формат: только XML (строгая типизация)
  • WSDL: Web Services Description Language — описание интерфейса сервиса
  • WS-Security: встроенная поддержка безопасности (шифрование, цифровые подписи)
  • Транзакции: поддержка распределенных транзакций (WS-AtomicTransaction)
SOAP vs REST:
  • SOAP: строгий стандарт, XML, сложнее, больше накладных расходов, лучше для enterprise (банки, платежи)
  • REST: архитектурный стиль, JSON/XML, проще, легковеснее, лучше для веб и мобильных приложений
5. Механизм NAT

NAT (Network Address Translation) — технология преобразования IP-адресов в заголовках IP-пакетов при прохождении через маршрутизатор. Позволяет устройствам с частными IP-адресами обращаться к интернету через один или несколько публичных IP-адресов.

Причины появления NAT:
  • Дефицит IPv4 адресов: всего 4.3 млрд адресов, недостаточно для всех устройств
  • Безопасность: скрытие внутренней структуры сети от внешнего мира
  • Гибкость: изменение внутренней адресации без изменения внешней
Типы NAT:
  • Static NAT (статический): постоянное отображение 1:1 между частным и публичным IP. Используется для серверов, которые должны быть доступны из интернета
  • Dynamic NAT (динамический): временное отображение из пула публичных адресов. Адрес выделяется на время сеанса
  • PAT/NAPT (Port Address Translation): много устройств используют один публичный IP через разные порты. Самый распространенный тип (используется в домашних роутерах)
Принцип работы PAT:
  • Устройство (192.168.1.5:3000) отправляет запрос на внешний сервер
  • NAT заменяет адрес источника на публичный (203.0.113.1:5000) и запоминает соответствие в таблице
  • Ответ приходит на 203.0.113.1:5000
  • NAT смотрит в таблицу и перенаправляет на 192.168.1.5:3000
Проблемы NAT:
  • Нарушение end-to-end принципа: устройства за NAT недоступны напрямую
  • Проблемы с P2P: сложности с прямым соединением (решение: STUN, TURN, UPnP)
  • Некоторые протоколы: FTP, SIP требуют специальной поддержки (ALG - Application Layer Gateway)

Решение: переход на IPv6 (128-битные адреса, достаточно для всех устройств), но NAT все еще широко используется.

6. Служба DNS

DNS (Domain Name System) — распределенная иерархическая система преобразования доменных имен в IP-адреса и обратно. Работает как "телефонная книга интернета".

Иерархия DNS:
  • Корневые серверы (Root): 13 логических серверов (a-m.root-servers.net), знают адреса серверов TLD
  • TLD серверы (Top-Level Domain): управляют доменами верхнего уровня (.com, .org, .ru, .uk)
  • Авторитетные серверы: хранят записи для конкретных доменов (google.com, yandex.ru)
  • Локальные DNS серверы: кэширующие серверы провайдера или организации
Процесс разрешения имени (рекурсивный запрос):
  • 1. Пользователь вводит google.com в браузере
  • 2. Проверка локального кэша браузера и ОС
  • 3. Запрос к локальному DNS серверу провайдера
  • 4. Если не в кэше: запрос к корневому серверу → получение адреса .com сервера
  • 5. Запрос к .com серверу → получение адреса NS для google.com
  • 6. Запрос к NS google.com → получение IP-адреса
  • 7. Возврат IP-адреса клиенту, кэширование на каждом уровне
Типы DNS записей:
  • A (Address): связывает имя с IPv4 адресом (example.com → 93.184.216.34)
  • AAAA: связывает имя с IPv6 адресом
  • CNAME (Canonical Name): псевдоним для другого имени (www.example.com → example.com)
  • MX (Mail Exchange): почтовые серверы для домена (приоритет + адрес)
  • NS (Name Server): авторитетные серверы имен для домена
  • TXT: текстовая информация (SPF, DKIM для почты, верификация домена)
  • PTR (Pointer): обратное преобразование IP → имя
  • SOA (Start of Authority): информация о зоне (основной сервер, email администратора, TTL)
Характеристики:
  • Порт: 53 (UDP для запросов, TCP для зонных трансферов)
  • TTL (Time To Live): время кэширования записи
  • Кэширование: на всех уровнях для ускорения и снижения нагрузки
7. Служба DHCP

DHCP (Dynamic Host Configuration Protocol) — протокол автоматической настройки сетевых параметров для устройств. Работает на прикладном уровне, использует UDP.

Процесс получения адреса (DORA):
  • 1. Discover (обнаружение): клиент отправляет широковещательный запрос (255.255.255.255) "Есть ли здесь DHCP сервер?"
  • 2. Offer (предложение): DHCP сервер отвечает предложением с IP-адресом, маской, шлюзом, DNS, временем аренды
  • 3. Request (запрос): клиент выбирает одно предложение (если серверов несколько) и отправляет запрос на его принятие
  • 4. Acknowledge (подтверждение): сервер подтверждает выделение адреса и отправляет финальную конфигурацию
Параметры, выдаваемые DHCP:
  • IP-адрес: уникальный адрес в сети
  • Маска подсети: определяет границы сети
  • Шлюз по умолчанию: маршрутизатор для выхода за пределы локальной сети
  • DNS серверы: адреса серверов для разрешения имен
  • Время аренды: срок действия выделенного адреса
  • Дополнительно: NTP серверы, WINS серверы, доменное имя
Типы выделения адресов:
  • Динамическое: адрес выдается на время из пула, после истечения аренды возвращается в пул
  • Автоматическое: адрес выдается навсегда из пула
  • Статическое (резервирование): привязка MAC-адреса к конкретному IP (для серверов, принтеров)
Преимущества:
  • Автоматизация настройки (не нужно вручную настраивать каждое устройство)
  • Централизованное управление адресами
  • Предотвращение конфликтов IP-адресов
  • Эффективное использование адресного пространства
  • Легкость изменения конфигурации сети

Порты: UDP 67 (сервер), UDP 68 (клиент). Relay Agent: для работы DHCP через маршрутизаторы (широковещательные пакеты не проходят через роутеры).

8. Протоколы TCP/UDP

TCP и UDP — два основных протокола транспортного уровня модели OSI, обеспечивающие передачу данных между приложениями.

TCP (Transmission Control Protocol):
  • Ориентирован на соединение: перед передачей данных устанавливается соединение (3-way handshake)
  • Надежность: гарантирует доставку всех пакетов в правильном порядке
  • Контроль ошибок: использует контрольные суммы, подтверждения (ACK), повторную передачу потерянных пакетов
  • Контроль потока: sliding window — предотвращает переполнение буфера получателя
  • Контроль перегрузки: slow start, congestion avoidance — адаптация к загрузке сети
  • Упорядочивание: пакеты нумеруются, собираются в правильном порядке
  • Full-duplex: двунаправленная передача данных
Установка соединения TCP (3-way handshake):
  • 1. SYN: клиент отправляет SYN пакет с начальным sequence number
  • 2. SYN-ACK: сервер отвечает SYN-ACK (подтверждение + свой sequence number)
  • 3. ACK: клиент отправляет ACK, соединение установлено
Завершение соединения TCP (4-way handshake):
  • FIN → ACK → FIN → ACK (каждая сторона закрывает свою половину соединения)
UDP (User Datagram Protocol):
  • Без установки соединения: просто отправляет датаграммы
  • Ненадежный: не гарантирует доставку, порядок, отсутствие дубликатов
  • Без контроля потока и перегрузки: отправляет с максимальной скоростью
  • Минимальные накладные расходы: заголовок всего 8 байт (у TCP 20+ байт)
  • Быстрее TCP: нет задержек на установку соединения и подтверждения
  • Поддержка multicast/broadcast: отправка одного пакета многим получателям
Когда использовать:
  • TCP: HTTP/HTTPS, FTP, SMTP, SSH, Telnet — когда важна надежность и целостность данных
  • UDP: DNS, DHCP, VoIP, видеостриминг, онлайн-игры, IoT — когда важна скорость, допустимы потери
Сравнение:
  • Скорость: UDP быстрее (нет накладных расходов)
  • Надежность: TCP надежнее (гарантии доставки)
  • Порядок: TCP сохраняет порядок, UDP нет
  • Размер заголовка: TCP 20-60 байт, UDP 8 байт
  • Применение: TCP для критичных данных, UDP для реального времени
9. Безопасность в сетях

Сетевая безопасность — комплекс мер для защиты данных, ресурсов и коммуникаций от несанкционированного доступа, изменения и уничтожения.

Основные принципы безопасности (CIA Triad):
  • Конфиденциальность (Confidentiality): защита информации от несанкционированного доступа. Реализуется через шифрование, контроль доступа, VPN. Пример: шифрование данных банковской карты при онлайн-покупке
  • Целостность (Integrity): гарантия того, что данные не были изменены при передаче или хранении. Реализуется через хеш-функции (MD5, SHA-256), цифровые подписи, контрольные суммы. Пример: проверка целостности скачанного файла по хешу
  • Доступность (Availability): обеспечение доступа к ресурсам для авторизованных пользователей. Защита от DDoS атак, резервирование, балансировка нагрузки
Механизмы безопасности:
  • Аутентификация (Authentication): проверка подлинности пользователя или системы. Методы: пароли, биометрия, токены, сертификаты, многофакторная аутентификация (2FA, MFA). Отвечает на вопрос "Кто ты?"
  • Авторизация (Authorization): предоставление прав доступа к ресурсам после аутентификации. Модели: DAC (дискреционная), MAC (мандатная), RBAC (ролевая), ABAC (атрибутная). Отвечает на вопрос "Что тебе разрешено?"
  • Невозможность отказа (Non-repudiation): невозможность отрицать совершенное действие (цифровые подписи, логи)
Криптография:
  • Криптография: наука о методах шифрования информации для защиты от несанкционированного доступа
  • Криптоанализ: наука о методах взлома шифров и криптосистем
  • Криптология: объединяет криптографию и криптоанализ
  • Шифр: алгоритм преобразования открытого текста в зашифрованный (Caesar, AES, RSA)
  • Код: замена целых слов или фраз на другие символы/слова (кодовые книги)
  • Ключ шифра: секретная информация, используемая алгоритмом для шифрования/дешифрования
Типы шифрования:
  • Симметричное: один ключ для шифрования и дешифрования (AES, DES, 3DES). Быстрое, но проблема обмена ключами
  • Асимметричное: пара ключей — публичный (шифрование) и приватный (дешифрование) (RSA, ECC). Медленнее, но безопасный обмен ключами
  • Гибридное: асимметричное для обмена симметричным ключом, затем симметричное для данных (используется в SSL/TLS)
Протоколы безопасности:
  • IPsec (IP Security): набор протоколов для защиты IP-пакетов на сетевом уровне. Режимы: транспортный (только данные), туннельный (весь пакет). Используется в VPN. Протоколы: AH (аутентификация), ESP (шифрование)
  • SSL/TLS (Secure Sockets Layer / Transport Layer Security): протоколы шифрования на транспортном уровне. TLS — современная версия SSL. Используется в HTTPS, FTPS, SMTPS. Обеспечивает: шифрование, аутентификацию сервера (сертификаты), целостность данных
  • HTTPS: HTTP + TLS/SSL. Защищает веб-трафик. Сертификаты выдаются Certificate Authority (CA): Let's Encrypt, DigiCert, Comodo
  • DNSsec (DNS Security Extensions): расширение DNS для защиты от подделки записей. Использует цифровые подписи для проверки подлинности DNS-ответов. Защищает от DNS spoofing, cache poisoning
Угрозы безопасности:
  • Перехват данных: прослушивание трафика (sniffing)
  • MITM (Man-in-the-Middle): перехват и изменение данных между двумя сторонами
  • DDoS: отказ в обслуживании через перегрузку
  • Фишинг: обман пользователей для получения данных
  • Вредоносное ПО: вирусы, трояны, ransomware
10. Модели взаимодействия сетевых приложений (OSI/ISO, TCP/IP)

Сетевые модели — концептуальные фреймворки, описывающие функции сетевых систем через уровни абстракции.

Модель OSI (Open Systems Interconnection):

Разработана ISO в 1984 году. 7 уровней, каждый выполняет определенные функции:

  • 7. Прикладной (Application): интерфейс для приложений. Протоколы: HTTP, FTP, SMTP, DNS, Telnet, SSH. Функции: передача файлов, электронная почта, удаленный доступ
  • 6. Представления (Presentation): преобразование данных, шифрование, сжатие. Форматы: JPEG, GIF, MPEG, SSL/TLS. Обеспечивает независимость от представления данных
  • 5. Сеансовый (Session): управление сеансами связи, синхронизация. Протоколы: NetBIOS, RPC, PPTP. Функции: установка, поддержание, завершение сеансов
  • 4. Транспортный (Transport): надежная доставка данных между приложениями. Протоколы: TCP, UDP, SCTP. Функции: сегментация, контроль ошибок, управление потоком, мультиплексирование портов
  • 3. Сетевой (Network): маршрутизация пакетов между сетями. Протоколы: IP, ICMP, ARP, OSPF, BGP. Функции: логическая адресация (IP), маршрутизация, фрагментация пакетов
  • 2. Канальный (Data Link): передача кадров между соседними узлами. Протоколы: Ethernet, Wi-Fi (802.11), PPP, HDLC. Функции: физическая адресация (MAC), обнаружение ошибок, управление доступом к среде (MAC sublayer)
  • 1. Физический (Physical): передача битов по физической среде. Стандарты: RS-232, USB, Ethernet кабели, оптоволокно. Функции: кодирование сигналов, модуляция, характеристики кабелей
Модель TCP/IP (Internet Protocol Suite):

Практическая модель, на которой построен интернет. 4 уровня:

  • 4. Прикладной (Application): объединяет уровни 5-7 OSI. Протоколы: HTTP, FTP, SMTP, DNS, SSH, Telnet, SNMP. Взаимодействие приложений с сетью
  • 3. Транспортный (Transport): аналогичен уровню 4 OSI. Протоколы: TCP (надежный), UDP (быстрый). Доставка данных между процессами, порты
  • 2. Интернет (Internet): аналогичен уровню 3 OSI. Протоколы: IP (IPv4, IPv6), ICMP, ARP, IGMP. Маршрутизация пакетов через сети
  • 1. Канальный (Link/Network Access): объединяет уровни 1-2 OSI. Протоколы: Ethernet, Wi-Fi, PPP. Передача данных в локальной сети
Сравнение моделей:
  • OSI: теоретическая, 7 уровней, более детальная, используется для обучения и стандартизации
  • TCP/IP: практическая, 4 уровня, на ней построен интернет, проще и гибче
  • Общее: обе используют концепцию уровней, инкапсуляции данных, стандартизированных протоколов
Инкапсуляция данных:
  • Прикладной: данные (Data)
  • Транспортный: сегмент/датаграмма (добавляется заголовок TCP/UDP с портами)
  • Сетевой: пакет (добавляется заголовок IP с адресами)
  • Канальный: кадр (добавляется заголовок Ethernet с MAC-адресами)
  • Физический: биты (передача по проводам/радио)

Принцип работы: каждый уровень добавляет свой заголовок при отправке (инкапсуляция) и удаляет при получении (декапсуляция). Уровни взаимодействуют только с соседними.

11. Архитектура распределенного приложения (клиент/сервер)

Архитектура клиент-сервер — модель взаимодействия в распределенных системах, где клиенты запрашивают услуги, а серверы их предоставляют.

Компоненты:
  • Клиент: инициирует запросы, имеет пользовательский интерфейс, выполняет презентационную логику, может работать офлайн (с кэшем). Примеры: веб-браузер, почтовый клиент, мобильное приложение
  • Сервер: ожидает и обрабатывает запросы, управляет ресурсами и данными, выполняет бизнес-логику, обеспечивает безопасность, работает 24/7. Примеры: веб-сервер (Apache, Nginx), сервер БД (PostgreSQL, MySQL), файловый сервер
  • Сеть: среда передачи данных между клиентом и сервером (LAN, WAN, интернет)
Типы архитектур:
  • Двухуровневая (2-tier): клиент напрямую взаимодействует с сервером БД. Клиент содержит бизнес-логику и UI. Проблемы: "толстый клиент", сложность обновления, безопасность. Пример: desktop приложение с прямым подключением к БД
  • Трехуровневая (3-tier): разделение на презентацию (клиент), бизнес-логику (сервер приложений), данные (сервер БД). Преимущества: разделение ответственности, легче масштабировать, безопаснее. Пример: веб-приложение (браузер → веб-сервер → БД)
  • Многоуровневая (N-tier): дополнительные уровни (кэш, очереди сообщений, API gateway, микросервисы). Пример: современные облачные приложения
Преимущества:
  • Централизованное управление: данные и логика на сервере, легче обновлять и обслуживать
  • Масштабируемость: можно добавлять серверы для обработки большего числа клиентов
  • Безопасность: контроль доступа на сервере, защита данных
  • Разделение ответственности: клиент — UI, сервер — данные и логика
  • Совместное использование ресурсов: один сервер обслуживает множество клиентов
Недостатки:
  • Единая точка отказа: если сервер падает, все клиенты теряют доступ
  • Узкое место: сервер может стать bottleneck при большой нагрузке
  • Зависимость от сети: без сети клиент не может работать
  • Стоимость: требуется мощное серверное оборудование
Варианты взаимодействия:
  • Тонкий клиент (Thin Client): минимальная логика на клиенте, все на сервере. Пример: веб-приложения
  • Толстый клиент (Thick Client): значительная логика на клиенте. Пример: desktop приложения
  • Гибридный: часть логики на клиенте, часть на сервере. Пример: мобильные приложения с офлайн режимом

Примеры: веб (браузер-сервер), email (Outlook-Exchange), файлы (проводник-файловый сервер), игры (клиент-игровой сервер).

12. Понятие сокета. Основные параметры сокета

Сокет (Socket) — программный интерфейс (API) для обмена данными между процессами через сеть. Это конечная точка двунаправленного канала связи. Концепция разработана в BSD Unix в начале 1980-х.

Определение:

Сокет — это абстракция, позволяющая приложениям отправлять и получать данные через сеть, используя стандартные операции ввода-вывода. Сокет идентифицируется комбинацией IP-адреса и номера порта.

Основные параметры сокета:
  • IP-адрес: идентифицирует узел в сети (например, 192.168.1.100 или 2001:db8::1). Определяет, на каком компьютере находится процесс
  • Порт: 16-битное число (0-65535) для идентификации приложения на узле. Известные порты (0-1023): HTTP-80, HTTPS-443, FTP-21, SSH-22. Зарегистрированные (1024-49151). Динамические (49152-65535)
  • Протокол: TCP или UDP, определяет способ передачи данных
  • Тип сокета:
    • SOCK_STREAM — потоковый сокет для TCP (надежный, упорядоченный, двунаправленный)
    • SOCK_DGRAM — датаграммный сокет для UDP (ненадежный, без соединения)
    • SOCK_RAW — сырой сокет для прямого доступа к IP (требует привилегий)
  • Семейство адресов: AF_INET (IPv4), AF_INET6 (IPv6), AF_UNIX (локальные сокеты)
Жизненный цикл TCP сокета:
  • Сервер:
    • 1. socket() — создание сокета
    • 2. bind() — привязка к IP-адресу и порту
    • 3. listen() — перевод в режим ожидания подключений
    • 4. accept() — принятие входящего соединения (блокирующий вызов)
    • 5. recv()/send() — прием и отправка данных
    • 6. close() — закрытие соединения
  • Клиент:
    • 1. socket() — создание сокета
    • 2. connect() — подключение к серверу
    • 3. send()/recv() — отправка и прием данных
    • 4. close() — закрытие соединения
Жизненный цикл UDP сокета:
  • socket() → bind() (для сервера) → sendto()/recvfrom() → close()
  • Нет connect(), listen(), accept() — соединение не устанавливается
Дополнительные параметры:
  • Буферы: размеры буферов отправки и приема (SO_SNDBUF, SO_RCVBUF)
  • Таймауты: время ожидания операций (SO_RCVTIMEO, SO_SNDTIMEO)
  • Опции: SO_REUSEADDR (повторное использование адреса), SO_KEEPALIVE (проверка соединения), TCP_NODELAY (отключение алгоритма Nagle)
  • Блокирующий/неблокирующий режим: определяет, будет ли операция ждать завершения

Применение: веб-серверы, чаты, онлайн-игры, потоковое видео, любое сетевое взаимодействие. Сокеты — основа всех сетевых приложений.

13. Виртуализация в РИС

Виртуализация — технология создания виртуальных (программных) версий физических ресурсов: серверов, хранилищ, сетей, операционных систем. Позволяет запускать несколько изолированных систем на одном физическом оборудовании.

Типы виртуализации:
  • 1. Аппаратная виртуализация (гипервизоры):
    • Тип 1 (Bare-metal): гипервизор работает напрямую на оборудовании. Примеры: VMware ESXi, Microsoft Hyper-V, Xen, KVM. Высокая производительность, используется в дата-центрах
    • Тип 2 (Hosted): гипервизор работает поверх ОС. Примеры: VMware Workstation, VirtualBox, Parallels. Удобен для разработки и тестирования
  • 2. Контейнеризация: изоляция на уровне ОС, общее ядро. Технологии: Docker, Kubernetes, LXC, Podman. Легковесные, быстрый запуск (секунды), меньше накладных расходов. Идеально для микросервисов
  • 3. Виртуализация приложений: изоляция приложений от ОС. Примеры: VMware ThinApp, Microsoft App-V. Приложение работает в изолированной среде
  • 4. Виртуализация рабочих столов (VDI): удаленный доступ к виртуальным рабочим столам. Примеры: Citrix Virtual Apps, VMware Horizon
  • 5. Виртуализация сети (SDN): программное управление сетью. Примеры: VMware NSX, OpenFlow, Cisco ACI
  • 6. Виртуализация хранилищ (SAN): объединение физических хранилищ в логические пулы
Преимущества виртуализации в РИС:
  • Эффективное использование ресурсов: один сервер может запускать 10-20 виртуальных машин, утилизация CPU до 80% вместо 10-15%
  • Изоляция: сбой одной VM не влияет на другие, безопасность
  • Быстрое развертывание: создание новой VM за минуты вместо дней на физический сервер
  • Миграция: живая миграция VM между серверами без остановки (vMotion, Live Migration)
  • Снимки (snapshots): сохранение состояния системы, откат при ошибках
  • Тестирование: создание изолированных сред для разработки и тестирования
  • Экономия: меньше физических серверов, электроэнергии, охлаждения, места
  • Отказоустойчивость: автоматический перезапуск VM на другом хосте при сбое
  • Масштабируемость: динамическое выделение ресурсов (CPU, RAM) по требованию
Применение в РИС:
  • Облачные вычисления: основа IaaS (AWS EC2, Azure VM)
  • Дата-центры: консолидация серверов, снижение затрат
  • Микросервисы: контейнеры для изоляции сервисов
  • CI/CD: автоматическое создание сред для тестирования
  • Disaster Recovery: быстрое восстановление из резервных копий
Контейнеры vs Виртуальные машины:
  • VM: полная ОС, гипервизор, изоляция на уровне оборудования, больше ресурсов, медленный старт (минуты)
  • Контейнеры: общее ядро ОС, изоляция на уровне процессов, меньше ресурсов, быстрый старт (секунды)
14. Синхронизация в РИС

Синхронизация — координация выполнения параллельных процессов для обеспечения корректного доступа к общим ресурсам и согласованности данных в распределенной системе.

Проблемы без синхронизации:
  • Гонки данных (Race Condition): результат зависит от порядка выполнения процессов. Пример: два процесса одновременно увеличивают счетчик, один результат теряется
  • Взаимоблокировки (Deadlock): процессы ждут друг друга бесконечно. Условия: взаимное исключение, удержание и ожидание, отсутствие вытеснения, циклическое ожидание
  • Голодание (Starvation): процесс никогда не получает доступ к ресурсу
  • Несогласованность данных: разные узлы видят разные версии данных
Механизмы синхронизации:
  • Мьютексы (Mutex - Mutual Exclusion): блокировка для взаимного исключения. Только один процесс может владеть мьютексом. Операции: lock() (захват), unlock() (освобождение). Используется для защиты критических секций
  • Семафоры (Semaphore): счетчик доступных ресурсов. Бинарный семафор (0 или 1) = мьютекс. Счетный семафор (N) — для ограничения числа одновременных доступов. Операции: wait()/P() (уменьшение), signal()/V() (увеличение)
  • Мониторы: высокоуровневая конструкция, объединяет данные и методы с автоматической синхронизацией. Только один поток может выполнять методы монитора. Условные переменные для ожидания событий
  • Барьеры (Barrier): точка синхронизации, где все процессы должны дождаться друг друга перед продолжением. Используется в параллельных вычислениях
  • Спинлоки (Spinlock): активное ожидание в цикле. Эффективны для коротких блокировок
  • Read-Write Locks: множественные читатели или один писатель. Повышает параллелизм
Синхронизация времени в РИС:
  • Проблема: часы на разных узлах идут с разной скоростью (clock drift), нет глобального времени
  • NTP (Network Time Protocol): протокол синхронизации времени через интернет. Точность до миллисекунд. Иерархия stratum (0-15)
  • Логические часы Лампорта: упорядочивание событий без физического времени. Каждый процесс имеет счетчик, увеличивает при событии, при получении сообщения берет max(свой, полученный)+1
  • Векторные часы: расширение часов Лампорта, каждый процесс хранит вектор счетчиков всех процессов. Позволяет определить причинно-следственные связи
  • Алгоритм Беркли: координатор собирает время со всех узлов, вычисляет среднее, рассылает корректировки
Распределенная синхронизация:
  • Распределенные блокировки: координация через центральный сервер, токены, или голосование
  • Алгоритм Ламберта: запрос разрешения у всех узлов, получение подтверждений
  • Алгоритм кольца: токен передается по кольцу, владелец токена может войти в критическую секцию
  • Консенсус: алгоритмы Paxos, Raft для согласования состояния между узлами
  • Двухфазная фиксация (2PC): координатор спрашивает участников (prepare), затем фиксирует (commit) или откатывает (abort)
Модели согласованности:
  • Строгая согласованность: все видят изменения мгновенно (невозможна в распределенных системах)
  • Последовательная согласованность: все видят операции в одном порядке
  • Причинная согласованность: связанные операции видны в правильном порядке
  • Eventual consistency: в конечном итоге все узлы придут к одному состоянию (используется в NoSQL)
15. Виды и типы РИС

РИС (Распределенные Информационные Системы) — системы, компоненты которых размещены на разных сетевых узлах и взаимодействуют через обмен сообщениями для достижения общей цели.

Классификация по архитектуре:
  • Клиент-сервер: централизованный сервер обслуживает множество клиентов. Примеры: веб-приложения, базы данных, файловые серверы. Преимущества: простота управления, безопасность. Недостатки: единая точка отказа, узкое место
  • Peer-to-peer (P2P): равноправные узлы, каждый может быть клиентом и сервером. Типы: чистый P2P (все равны), гибридный (есть супер-узлы). Примеры: BitTorrent, Skype, блокчейн. Преимущества: масштабируемость, отказоустойчивость. Недостатки: сложность управления, безопасность
  • Многоуровневая (N-tier): разделение на слои (презентация, бизнес-логика, данные, интеграция). Примеры: enterprise приложения. Преимущества: модульность, масштабируемость каждого уровня
  • Микросервисы: приложение из независимых сервисов, каждый со своей БД. Примеры: Netflix, Amazon, Uber. Преимущества: независимое развертывание, технологическое разнообразие. Недостатки: сложность координации, распределенные транзакции
Классификация по назначению:
  • Распределенные базы данных: данные распределены по узлам. Типы: репликация (копии на разных узлах), партиционирование/шардинг (разные данные на разных узлах). Примеры: Cassandra, MongoDB, CockroachDB
  • Распределенные файловые системы: хранение файлов на множестве узлов. Примеры: HDFS (Hadoop), GFS (Google), Ceph, GlusterFS. Используются для Big Data
  • Распределенные вычислительные системы: параллельная обработка задач. Примеры: кластеры (HPC), грид-системы, MapReduce, Spark
  • Распределенные системы обмена сообщениями: асинхронная коммуникация. Примеры: RabbitMQ, Apache Kafka, ActiveMQ
  • Распределенные кэши: ускорение доступа к данным. Примеры: Redis, Memcached, Hazelcast
Классификация по связности:
  • Слабосвязанные (Loosely Coupled): компоненты независимы, взаимодействуют через стандартные интерфейсы (REST API, очереди сообщений). Легче масштабировать и изменять. Примеры: микросервисы, SOA
  • Сильносвязанные (Tightly Coupled): компоненты зависят друг от друга, прямые вызовы. Сложнее масштабировать. Примеры: монолитные приложения с распределенными модулями
Классификация по синхронности:
  • Синхронные: клиент ждет ответа (RPC, REST API). Проще программировать, но блокирующие операции
  • Асинхронные: клиент не ждет ответа (очереди сообщений, pub/sub). Выше производительность, но сложнее отладка
Классификация по масштабу:
  • Локальные (LAN): в пределах одного здания/кампуса
  • Глобальные (WAN): через интернет, между городами/странами
  • Облачные: в дата-центрах провайдеров (AWS, Azure, GCP)
Специальные типы:
  • Grid Computing: объединение гетерогенных ресурсов для решения сложных задач
  • Cluster Computing: группа однородных компьютеров, работающих как единая система
  • Edge Computing: обработка данных на краю сети, ближе к источнику (IoT)
  • Fog Computing: промежуточный слой между облаком и устройствами
16. Преимущества распределенных вычислений

Распределенные вычисления предоставляют множество преимуществ по сравнению с централизованными системами:

1. Масштабируемость (Scalability):
  • Горизонтальное масштабирование: добавление новых узлов для увеличения мощности (scale-out). Легче и дешевле, чем вертикальное
  • Вертикальное масштабирование: увеличение ресурсов узла (scale-up). Ограничено физическими пределами
  • Эластичность: автоматическое добавление/удаление ресурсов по требованию (облака)
  • Пример: добавить 10 серверов проще, чем купить один в 10 раз мощнее
2. Производительность:
  • Параллельная обработка: задачи выполняются одновременно на разных узлах
  • Снижение латентности: размещение ресурсов ближе к пользователям (CDN, edge computing)
  • Балансировка нагрузки: распределение запросов между серверами
  • Пример: обработка 1 млрд записей за 1 час на 1000 узлах вместо 1000 часов на одном
3. Надежность и отказоустойчивость:
  • Отсутствие единой точки отказа: сбой одного узла не останавливает систему
  • Репликация данных: копии на разных узлах защищают от потери
  • Автоматическое восстановление: перезапуск задач на других узлах
  • Graceful degradation: система продолжает работать с ограниченной функциональностью
  • Пример: если 1 из 100 серверов упал, 99% мощности остается
4. Доступность (Availability):
  • Высокая доступность (HA): 99.9% (8.76 часов простоя в год), 99.99% (52 минуты), 99.999% (5 минут)
  • Географическое распределение: данные доступны из разных регионов
  • Резервирование: активные и пассивные резервные узлы
  • Пример: сайт доступен даже если дата-центр в одной стране недоступен
5. Экономичность:
  • Commodity hardware: использование обычных серверов вместо дорогих суперкомпьютеров
  • Оплата по факту: в облаках платишь только за используемые ресурсы
  • Совместное использование: один ресурс обслуживает множество пользователей
  • Пример: 100 обычных серверов дешевле одного мейнфрейма с той же мощностью
6. Географическое распределение:
  • Близость к пользователям: снижение задержек, лучший UX
  • Соответствие законам: хранение данных в определенных юрисдикциях (GDPR)
  • Disaster Recovery: защита от региональных катастроф
  • Пример: CDN размещает контент в 200+ городах мира
7. Гибкость и модульность:
  • Независимое развитие: компоненты можно обновлять отдельно
  • Технологическое разнообразие: разные языки и технологии для разных задач
  • Изоляция сбоев: проблема в одном модуле не влияет на другие
8. Совместная работа:
  • Распределенные команды: разработчики в разных местах работают над одной системой
  • Совместный доступ к данным: пользователи из разных мест работают с общими ресурсами
  • Пример: Google Docs — одновременное редактирование документа
17. Использование распределенных вычислений

Распределенные вычисления применяются в широком спектре областей для решения сложных задач, требующих больших вычислительных ресурсов или высокой доступности.

1. Научные исследования:
  • Моделирование климата: прогнозирование погоды, изучение глобального потепления (ECMWF, NOAA)
  • Анализ генома: секвенирование ДНК, поиск генетических маркеров (Human Genome Project)

  • Молекулярное моделирование: разработка лекарств, изучение белков (Folding@home)
2. Big Data и аналитика:
  • Обработка больших данных: Hadoop, Spark для анализа петабайтов данных
  • Data Mining: поиск закономерностей в больших массивах данных
  • Real-time analytics: анализ потоковых данных (Apache Kafka, Flink)
  • Рекомендательные системы: Netflix, Amazon, YouTube анализируют поведение миллионов пользователей
  • Пример: обработка 500 ТБ логов в день для анализа поведения пользователей
3. Веб-сервисы и приложения:
  • Поисковые системы: Google индексирует триллионы страниц на тысячах серверов
  • Социальные сети: Facebook, Twitter обрабатывают миллиарды запросов в день
  • E-commerce: Amazon, Alibaba — миллионы транзакций, рекомендации, инвентаризация
  • Стриминг: Netflix, YouTube доставляют видео через CDN по всему миру
  • Облачные сервисы: Gmail, Office 365, Dropbox
4. Финансовый сектор:
  • Высокочастотный трейдинг (HFT): анализ рынков и совершение сделок за микросекунды
  • Обработка транзакций: банковские системы обрабатывают миллионы платежей
  • Риск-анализ: моделирование финансовых рисков, стресс-тестирование
  • Обнаружение мошенничества: анализ транзакций в реальном времени
  • Блокчейн: распределенные реестры для криптовалют (Bitcoin, Ethereum)
5. Машинное обучение и AI:
  • Обучение нейронных сетей: распределенное обучение на GPU кластерах (TensorFlow, PyTorch)
  • Компьютерное зрение: распознавание лиц, объектов, медицинская диагностика
  • Обработка естественного языка: ChatGPT, переводчики, голосовые ассистенты
  • Автономные автомобили: обработка данных сенсоров, обучение моделей
6. Рендеринг и графика:
  • Кино и анимация: Pixar, DreamWorks используют рендер-фермы для создания фильмов
  • Видеообработка: кодирование, транскодирование видео (YouTube, Twitch)
  • Игры: серверы для многопользовательских онлайн-игр (MMO)
7. Интернет вещей (IoT):
  • Умные города: управление трафиком, освещением, коммунальными услугами
  • Промышленный IoT: мониторинг оборудования, предиктивное обслуживание
  • Умный дом: координация устройств, обработка данных сенсоров
8. Контент и медиа:
  • CDN (Content Delivery Network): Cloudflare, Akamai доставляют контент из ближайших точек
  • Стриминг музыки: Spotify, Apple Music
  • Облачный гейминг: Google Stadia, GeForce Now
9. Телекоммуникации:
  • 5G сети: распределенная обработка для низкой латентности
  • VoIP: Skype, Zoom, WhatsApp звонки

10. Добровольные вычисления:

  • BOINC: платформа для научных проектов
18. Параллельные/сетевые вычисления

Параллельные и сетевые вычисления — два подхода к распределению вычислительной нагрузки, имеющие разные характеристики и области применения.

Параллельные вычисления:

Определение: одновременное выполнение нескольких вычислительных задач на множестве процессоров или ядер.

  • Типы параллелизма:
    • Параллелизм данных (Data Parallelism): одна операция применяется к разным частям данных одновременно. Пример: обработка каждого пикселя изображения на отдельном ядре
    • Параллелизм задач (Task Parallelism): разные операции выполняются одновременно. Пример: один поток читает файл, другой обрабатывает данные, третий записывает результат
    • Конвейерный параллелизм (Pipeline): задача разбита на этапы, каждый этап обрабатывает свою часть. Пример: процессор с конвейером команд
  • Архитектуры:
    • Многоядерные процессоры: несколько ядер на одном чипе (Intel Core i7, AMD Ryzen)
    • Многопроцессорные системы (SMP): несколько процессоров с общей памятью
    • GPU (Graphics Processing Unit): тысячи простых ядер для массивно-параллельных вычислений (CUDA, OpenCL)
    • Кластеры: группа компьютеров, соединенных высокоскоростной сетью (InfiniBand)
    • Суперкомпьютеры: тысячи узлов для научных расчетов (Top500)
  • Модели программирования:
    • Общая память: потоки разделяют адресное пространство (OpenMP, pthreads)
    • Передача сообщений: процессы обмениваются данными через сообщения (MPI - Message Passing Interface)
    • SIMD: Single Instruction Multiple Data — одна инструкция для множества данных (векторизация)
Сетевые вычисления:

Определение: вычисления на узлах, связанных через сеть (обычно интернет), часто географически распределенных.

  • Характеристики:
    • Узлы соединены через LAN/WAN/интернет
    • Высокая латентность связи (миллисекунды-секунды)
    • Гетерогенность: разные ОС, архитектуры, производительность
    • Автономность узлов: могут присоединяться и отключаться
    • Географическое распределение
  • Типы:
    • Grid Computing: объединение гетерогенных ресурсов организаций для решения задач. Middleware: Globus Toolkit. Пример: научные грид-сети для обработки данных LHC
    • Volunteer Computing: использование простаивающих ресурсов добровольцев. Платформа: BOINC. Примеры: SETI@home, Folding@home
    • Cloud Computing: вычисления в дата-центрах провайдеров. Примеры: AWS, Azure, GCP
  • Модели:
    • Master-Worker: мастер распределяет задачи, воркеры выполняют
    • MapReduce: map (обработка) + reduce (агрегация). Hadoop, Spark
    • Bag of Tasks: независимые задачи, выполняются на доступных узлах
Сравнение параллельных и сетевых вычислений:
  • Связь: параллельные — быстрая (наносекунды-микросекунды), сетевые — медленная (миллисекунды-секунды)
  • Расположение: параллельные — один узел/кластер, сетевые — географически распределены
  • Однородность: параллельные — обычно однородные, сетевые — гетерогенные
  • Синхронизация: параллельные — тесная, сетевые — слабая
  • Отказоустойчивость: параллельные — ниже, сетевые — выше (больше узлов)
  • Задачи: параллельные — тесно связанные (матричные операции), сетевые — слабо связанные (обработка файлов)
Закон Амдала:

Ускорение = 1 / ((1 - P) + P/N), где P — доля параллелизуемого кода, N — число процессоров. Показывает, что последовательная часть ограничивает ускорение.

Применение:
  • Параллельные: научные симуляции, рендеринг, обучение ML, обработка сигналов
  • Сетевые: веб-сервисы, Big Data, добровольные вычисления, облачные приложения
19. Облачные РИС

Облачные вычисления (Cloud Computing) — модель предоставления вычислительных ресурсов (серверы, хранилища, приложения) через интернет по требованию с оплатой по факту использования.

Характеристики облачных вычислений (NIST):
  • Самообслуживание по требованию: пользователь может получить ресурсы автоматически без взаимодействия с провайдером
  • Широкий сетевой доступ: доступ через стандартные механизмы (браузер, API) с любых устройств
  • Пул ресурсов: ресурсы провайдера объединены для обслуживания множества клиентов (мультитенантность)
  • Эластичность: быстрое масштабирование вверх и вниз по требованию, автоматически
  • Измеримый сервис: использование ресурсов отслеживается, контролируется, оплачивается
Модели обслуживания (Service Models):
  • IaaS (Infrastructure as a Service):
    • Предоставление: виртуальные машины, сети, хранилища
    • Контроль: пользователь управляет ОС, приложениями, данными
    • Примеры: AWS EC2, Azure Virtual Machines, Google Compute Engine, DigitalOcean
    • Использование: полный контроль, миграция legacy приложений
  • PaaS (Platform as a Service):
    • Предоставление: платформа для разработки и развертывания приложений
    • Контроль: пользователь управляет приложениями и данными, провайдер — инфраструктурой
    • Примеры: Google App Engine, Heroku, AWS Elastic Beanstalk, Azure App Service
    • Использование: быстрая разработка, фокус на коде, не на инфраструктуре
  • SaaS (Software as a Service):
    • Предоставление: готовые приложения через браузер
    • Контроль: пользователь только использует приложение
    • Примеры: Gmail, Office 365, Salesforce, Dropbox, Slack, Zoom
    • Использование: нет необходимости в установке и обслуживании
  • Дополнительные модели:
    • FaaS/Serverless: выполнение функций без управления серверами (AWS Lambda, Azure Functions)
    • DBaaS: управляемые базы данных (AWS RDS, Azure SQL Database)
    • CaaS: контейнеры как сервис (AWS ECS, Google Kubernetes Engine)
Модели развертывания (Deployment Models):
  • Публичное облако (Public Cloud): инфраструктура провайдера, доступна всем. Примеры: AWS, Azure, GCP. Преимущества: низкая стоимость, масштабируемость. Недостатки: меньше контроля, вопросы безопасности
  • Частное облако (Private Cloud): инфраструктура для одной организации, может быть on-premise или у провайдера. Примеры: VMware vCloud, OpenStack. Преимущества: больше контроля, безопасность. Недостатки: выше стоимость
  • Гибридное облако (Hybrid Cloud): комбинация публичного и частного. Критичные данные в частном, остальное в публичном. Примеры: Azure Arc, AWS Outposts
  • Мультиоблако (Multi-Cloud): использование нескольких провайдеров. Избежание vendor lock-in, оптимизация стоимости
  • Общественное облако (Community Cloud): для группы организаций с общими требованиями (например, государственные учреждения)
Преимущества облачных РИС:
  • Оплата по факту (Pay-as-you-go): платишь только за использованные ресурсы, нет капитальных затрат
  • Эластичность: автоматическое масштабирование под нагрузку
  • Глобальная доступность: дата-центры по всему миру, низкая латентность
  • Высокая доступность: SLA 99.9-99.99%, резервирование
  • Отсутствие обслуживания: провайдер управляет инфраструктурой
  • Быстрое развертывание: ресурсы за минуты вместо недель
  • Инновации: доступ к новейшим технологиям (AI, ML, IoT)
Недостатки и риски:
  • Зависимость от провайдера (Vendor Lock-in): сложность миграции
  • Безопасность и приватность: данные у третьей стороны
  • Зависимость от интернета: без сети нет доступа
  • Соответствие законам: требования к хранению данных (GDPR, HIPAA)
  • Непредсказуемые затраты: при неправильной настройке
Основные провайдеры:
  • AWS (Amazon Web Services): лидер рынка, 200+ сервисов
  • Microsoft Azure: интеграция с Microsoft продуктами
  • Google Cloud Platform: сильны в ML/AI, Kubernetes
  • Alibaba Cloud: лидер в Азии
  • Yandex Cloud, Mail.ru Cloud: российские провайдеры
20. Виды системных архитектур РИС

Системная архитектура определяет структуру распределенной системы, способы взаимодействия компонентов и распределение ответственности.

1. Клиент-сервер (Client-Server):
  • Описание: централизованный сервер предоставляет услуги множеству клиентов
  • Варианты: 2-tier (клиент-БД), 3-tier (клиент-сервер приложений-БД), N-tier
  • Преимущества: централизованное управление, безопасность, простота
  • Недостатки: единая точка отказа, узкое место на сервере
  • Примеры: веб-приложения, email, файловые серверы
2. Peer-to-Peer (P2P):
  • Описание: равноправные узлы, каждый может быть клиентом и сервером
  • Типы:
    • Чистый P2P: все узлы равны, нет центрального сервера (Gnutella)
    • Гибридный P2P: есть центральный индекс или супер-узлы (BitTorrent с трекерами, Skype)
    • Структурированный P2P: организованная топология, DHT (Distributed Hash Table) для поиска (Chord, Kademlia)
  • Преимущества: масштабируемость, отказоустойчивость, нет единой точки отказа
  • Недостатки: сложность управления, безопасность, поиск ресурсов
  • Примеры: BitTorrent, блокчейн, IPFS
3. Многоуровневая архитектура (N-tier):
  • Описание: разделение системы на логические уровни
  • Уровни:
    • Презентация: пользовательский интерфейс (веб, мобильное приложение)
    • Бизнес-логика: обработка запросов, правила (сервер приложений)
    • Данные: хранение и управление данными (СУБД)
    • Интеграция: взаимодействие с внешними системами (API gateway, ESB)
  • Преимущества: разделение ответственности, независимое масштабирование уровней, модульность
  • Примеры: enterprise приложения, банковские системы
4. Микросервисы (Microservices):
  • Описание: приложение состоит из множества небольших независимых сервисов
  • Характеристики:
    • Каждый сервис — отдельный процесс с собственной БД
    • Взаимодействие через легковесные протоколы (REST, gRPC, очереди)
    • Независимое развертывание и масштабирование
    • Технологическое разнообразие (разные языки, БД)
    • Организация вокруг бизнес-возможностей
  • Преимущества: гибкость, масштабируемость, отказоустойчивость, быстрая разработка
  • Недостатки: сложность координации, распределенные транзакции, мониторинг
  • Паттерны: API Gateway, Service Discovery, Circuit Breaker, Saga
  • Примеры: Netflix, Amazon, Uber, Spotify
5. Service-Oriented Architecture (SOA):
  • Описание: приложение из слабосвязанных сервисов с четкими интерфейсами
  • Компоненты: ESB (Enterprise Service Bus) для интеграции, SOAP/WSDL для взаимодействия
  • Отличие от микросервисов: SOA — более монолитные сервисы, централизованная интеграция (ESB), микросервисы — мелкие сервисы, децентрализованная интеграция
  • Примеры: корпоративные системы интеграции
6. Event-Driven Architecture (EDA):
  • Описание: компоненты реагируют на события асинхронно
  • Паттерны:
    • Pub/Sub: издатели публикуют события, подписчики получают (Kafka, RabbitMQ)
    • Event Sourcing: состояние хранится как последовательность событий
    • CQRS: разделение команд (запись) и запросов (чтение)
  • Преимущества: слабая связанность, масштабируемость, реактивность
  • Недостатки: сложность отладки, eventual consistency
  • Примеры: системы реального времени, IoT, финансовые системы
7. Grid Computing:
  • Описание: объединение гетерогенных вычислительных ресурсов разных организаций
  • Характеристики: географическое распределение, разные административные домены, middleware для координации
  • Middleware: Globus Toolkit, gLite
  • Примеры: научные грид-сети (WLCG для CERN)
8. Cluster Computing:
  • Описание: группа однородных компьютеров, работающих как единая система
  • Типы:
    • High Availability (HA): отказоустойчивость, автоматическое переключение
    • Load Balancing: распределение нагрузки между узлами
    • High Performance (HPC): параллельные вычисления для научных задач
  • Примеры: Beowulf кластеры, суперкомпьютеры
9. Serverless / FaaS (Function as a Service):
  • Описание: выполнение функций без управления серверами
  • Характеристики:
    • Автоматическое масштабирование до нуля
    • Оплата за время выполнения (миллисекунды)
    • Event-driven триггеры (HTTP, очереди, таймеры)
    • Stateless функции
  • Преимущества: нет управления инфраструктурой, автоматическое масштабирование, низкая стоимость для спорадических нагрузок
  • Недостатки: холодный старт, ограничения времени выполнения, vendor lock-in
  • Примеры: AWS Lambda, Azure Functions, Google Cloud Functions
10. Edge Computing:
  • Описание: обработка данных на краю сети, ближе к источнику
  • Назначение: снижение латентности, экономия пропускной способности, работа при отсутствии связи
  • Применение: IoT, автономные автомобили, AR/VR, 5G
  • Примеры: AWS Greengrass, Azure IoT Edge, Cloudflare Workers
11. Fog Computing:
  • Описание: промежуточный слой между облаком и устройствами
  • Отличие от Edge: Fog — распределенная инфраструктура на разных уровнях, Edge — на самом краю
  • Применение: умные города, промышленный IoT
Выбор архитектуры:

  • Микросервисы: сложные приложения, быстрая разработка, масштабируемость
  • Event-Driven: реактивные системы, реальное время
  • Serverless: спорадические нагрузки, быстрое прототипирование
  • P2P: распределение контента, блокчейн
