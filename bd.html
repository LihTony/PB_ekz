📚 Подготовка к экзамену: Базы данных

Вопрос: Понятие данных и баз данных. Виды баз данных. История развития баз данных.
Данные — это факты, сведения, представленные в формализованном виде для обработки. Отличаются от информации тем, что информация — это обработанные и интерпретированные данные.
База данных (БД) — организованная совокупность данных, хранящихся в структурированном виде и управляемых СУБД. БД обеспечивает централизованное хранение, минимизацию избыточности, многопользовательский доступ и целостность данных.

Виды БД по модели данных:
• Иерархические — древовидная структура (родитель-потомок), пример: IMS.
• Сетевые — граф с множественными связями, пример: IDMS.
• Реляционные — данные в таблицах, связи через ключи, пример: MySQL, PostgreSQL, Oracle.
• Объектно-ориентированные — хранение объектов с методами, пример: db4o.
• NoSQL: документные (MongoDB), ключ-значение (Redis), графовые (Neo4j), колоночные (Cassandra).

История развития:
• 1960-е: файловые системы, иерархические (IMS от IBM) и сетевые модели (CODASYL).
• 1970: Эдгар Кодд публикует статью о реляционной модели.
• 1980-е: коммерческие РСУБД (Oracle, DB2, Ingres), стандарт SQL.
• 1990-е: объектно-ориентированные и объектно-реляционные БД, распределенные системы.
• 2000-е: рост NoSQL для веб-приложений и больших данных (Google BigTable, Amazon Dynamo).
• 2010-е: NewSQL, облачные БД, in-memory БД.
Вопрос: Понятие реляционной модели данных. Преимущества и недостатки реляционной модели данных. Принципы построения реляционных баз данных.
Реляционная модель — данные представлены в виде таблиц (отношений), связанных между собой.

Преимущества: простота, гибкость, независимость данных, математическая основа, стандартизация (SQL), целостность данных.
Недостатки: сложность для иерархических структур, производительность при больших объемах, избыточность при нормализации.

Принципы: данные в таблицах; каждая таблица имеет уникальный ключ; связи через внешние ключи; атомарность значений; отсутствие дубликатов строк.
Вопрос: Понятие отношения. Основные компоненты отношения. Понятие ключа. Виды ключей. Понятие связи. Виды связей и их свойства.
Отношение (Relation) — таблица с уникальным именем, математически определяется как подмножество декартова произведения доменов. Состоит из заголовка (схемы) и тела (набора кортежей).

Основные компоненты отношения:
• Схема отношения — структура таблицы (имя + список атрибутов с доменами).
• Кортеж (tuple) — строка таблицы, экземпляр сущности.
• Атрибут — столбец таблицы, характеристика сущности.
• Домен — множество допустимых значений атрибута.
• Степень (degree) — количество атрибутов в отношении.
• Кардинальность (cardinality) — количество кортежей в отношении.
• Значение атрибута — конкретное значение в ячейке таблицы.

Ключ — минимальный набор атрибутов, однозначно идентифицирующий кортеж.

Виды ключей:
• Потенциальный ключ (candidate key) — любой атрибут или набор атрибутов, способный уникально идентифицировать строку.
• Первичный ключ (PRIMARY KEY) — выбранный потенциальный ключ; не может быть NULL; уникален.
• Внешний ключ (FOREIGN KEY) — атрибут, ссылающийся на первичный ключ другой таблицы.
• Альтернативный ключ — потенциальный ключ, не выбранный как первичный.
• Составной ключ — ключ из нескольких атрибутов.
• Суррогатный ключ — искусственный ключ (обычно автоинкремент), не имеющий бизнес-смысла.
• Естественный ключ — ключ с бизнес-смыслом (например, ИНН, номер паспорта).

Связь (Relationship) — ассоциация между таблицами, реализуемая через ключи.

Виды связей:
• Один-к-одному (1:1): одной записи в таблице A соответствует максимум одна запись в таблице B. Пример: Человек — Паспорт.
• Один-ко-многим (1:N): одной записи в таблице A соответствует несколько записей в таблице B. Пример: Отдел — Сотрудники.
• Многие-ко-многим (M:N): множественные связи с обеих сторон. Реализуется через промежуточную таблицу. Пример: Студенты — Курсы.

Свойства связей:
• Кардинальность — количественная характеристика связи (1:1, 1:N, M:N).
• Обязательность (модальность) — обязательное или необязательное участие сущности в связи (может ли быть NULL).
Вопрос: Организация связей. Ссылочная целостность.
Организация связей:
• 1:1 — внешний ключ в любой из таблиц с ограничением UNIQUE.
• 1:N — внешний ключ на стороне "многие", ссылающийся на первичный ключ стороны "один".
• M:N — промежуточная (связующая) таблица с двумя внешними ключами, ссылающимися на обе связываемые таблицы.

Пример M:N (Студенты-Курсы):
CREATE TABLE Students (student_id INT PRIMARY KEY, name VARCHAR(100));
CREATE TABLE Courses (course_id INT PRIMARY KEY, title VARCHAR(100));
CREATE TABLE Enrollments (
  student_id INT,
  course_id INT,
  PRIMARY KEY (student_id, course_id),
  FOREIGN KEY (student_id) REFERENCES Students(student_id),
  FOREIGN KEY (course_id) REFERENCES Courses(course_id)
);
Ссылочная целостность (Referential Integrity) — правило, гарантирующее, что значение внешнего ключа либо NULL, либо существует в связанной таблице как первичный ключ. Предотвращает "висячие ссылки".

Действия при нарушении:
• CASCADE — каскадное удаление/обновление связанных записей.
• SET NULL — установка NULL во внешнем ключе.
• SET DEFAULT — установка значения по умолчанию.
• RESTRICT / NO ACTION — запрет операции, вызывающей нарушение.

Пример:
FOREIGN KEY (department_id) REFERENCES Departments(id)
  ON DELETE CASCADE
  ON UPDATE CASCADE
Вопрос: Особенности проектирования и реализации связей на различных уровнях проектирования баз данных.
Инфологический уровень: связи отображаются как линии между сущностями с указанием кардинальности (ER-диаграммы).
Даталогический уровень: связи реализуются через первичные и внешние ключи; M:N преобразуются в промежуточные таблицы.
Физический уровень: создаются индексы на внешних ключах для оптимизации JOIN-операций; настраиваются правила ссылочной целостности.
Вопрос: Понятие аномалии операций с базами данных. Виды аномалий операций с базами данных и способы их устранения.
Аномалия — нежелательное явление при операциях с БД из-за неправильной структуры, приводящее к избыточности, несогласованности или потере данных.

Виды аномалий:

• Аномалия вставки (Insert Anomaly): невозможность добавить данные без других обязательных данных.
Пример: Таблица (Сотрудник, Отдел, Адрес_отдела). Нельзя добавить новый отдел без сотрудника.

• Аномалия удаления (Delete Anomaly): потеря важных данных при удалении строки.
Пример: При увольнении последнего сотрудника отдела теряется информация об адресе отдела.

• Аномалия обновления (Update Anomaly): необходимость изменения данных в нескольких местах, риск несогласованности при частичном обновлении.
Пример: Адрес отдела хранится в каждой строке сотрудника. При изменении адреса нужно обновить все строки, иначе возникнет несогласованность.

Способы устранения:
• Нормализация — декомпозиция таблиц на более мелкие для устранения избыточности и зависимостей.
• Разделение данных: создание отдельных таблиц для независимых сущностей.
• Правильная организация ключей и связей между таблицами.

Решение примера: Разделить на таблицы Employees (id, name, department_id) и Departments (id, name, address).
Вопрос: Понятие нормализации. Требования нормализации.
Нормализация — процесс организации данных для минимизации избыточности и устранения аномалий путем декомпозиции таблиц на более мелкие, связанные между собой.

Цели нормализации: устранение избыточности данных, предотвращение аномалий вставки/обновления/удаления, обеспечение целостности данных, упрощение поддержки БД.

Требования:
• Декомпозиция без потерь — возможность восстановления исходных данных через JOIN без появления лишних или потери существующих кортежей.
• Сохранение зависимостей — все функциональные зависимости должны быть представлены в результирующих таблицах.
• Последовательное приведение к нормальным формам — поэтапный процесс: 1NF → 2NF → 3NF → BCNF и выше.

Пример нормализации:
До нормализации (ненормализованная таблица):
Заказы (Заказ_ID, Клиент, Адрес_клиента, Товар1, Цена1, Товар2, Цена2)
После нормализации (3NF):
• Клиенты (Клиент_ID, Имя, Адрес)
• Заказы (Заказ_ID, Клиент_ID, Дата)
• Товары (Товар_ID, Название, Цена)
• Позиции_заказа (Заказ_ID, Товар_ID, Количество)
Вопрос: Теория зависимостей.
Функциональная зависимость (ФЗ): атрибут B функционально зависит от A, если каждому значению A соответствует единственное значение B. Обозначение: A → B.
Пример: Студент_ID → ФИО (по ID студента однозначно определяется его ФИО).

Полная функциональная зависимость: атрибут зависит от всего составного ключа целиком, а не от его части.
Пример: (Студент_ID, Курс_ID) → Оценка — полная зависимость. Но (Студент_ID, Курс_ID) → ФИО_студента — частичная (зависит только от Студент_ID).

Транзитивная функциональная зависимость: если A → B и B → C, то A → C (через промежуточный атрибут).
Пример: Студент_ID → Группа_ID и Группа_ID → Куратор, следовательно Студент_ID → Куратор.

Многозначная зависимость (MVD): одному значению атрибута A соответствует множество значений атрибута B независимо от других атрибутов. Обозначение: A →→ B.
Пример: Преподаватель →→ Телефоны и Преподаватель →→ Курсы (независимо друг от друга).

Зависимость соединения (Join Dependency): отношение можно без потерь декомпозировать на несколько проекций, которые затем соединяются обратно.

Аксиомы Армстронга (правила вывода ФЗ):
• Рефлексивность: если B ⊆ A, то A → B.
• Дополнение: если A → B, то AC → BC.
• Транзитивность: если A → B и B → C, то A → C.
Вопрос: Нормальные формы низких порядков.
1NF (Первая нормальная форма):
• Все атрибуты атомарны (неделимы, скалярные значения).
• Нет повторяющихся групп или массивов.
• Есть первичный ключ.
• Порядок строк не имеет значения.
Нарушение 1NF: Студент (ID, ФИО, Телефоны: "123, 456, 789") — телефоны не атомарны.
Приведение к 1NF: разделить на Студент (ID, ФИО) и Телефоны (ID, Студент_ID, Телефон).

2NF (Вторая нормальная форма):
• Соответствует 1NF.
• Все неключевые атрибуты полностью функционально зависят от первичного ключа (устранение частичных зависимостей).
• Актуально для составных ключей.
Нарушение 2NF: Заказы (Заказ_ID, Товар_ID, Название_товара, Количество) — Название_товара зависит только от Товар_ID, а не от всего ключа (Заказ_ID, Товар_ID).
Приведение к 2NF: Заказы (Заказ_ID, Товар_ID, Количество) и Товары (Товар_ID, Название_товара).

3NF (Третья нормальная форма):
• Соответствует 2NF.
• Отсутствуют транзитивные зависимости неключевых атрибутов от первичного ключа.
• Каждый неключевой атрибут зависит только от ключа, напрямую.
Нарушение 3NF: Сотрудники (ID, ФИО, Отдел_ID, Название_отдела) — Название_отдела зависит от Отдел_ID (транзитивная зависимость: ID → Отдел_ID → Название_отдела).
Приведение к 3NF: Сотрудники (ID, ФИО, Отдел_ID) и Отделы (Отдел_ID, Название_отдела).
Вопрос: Нормальные формы высоких порядков.
BCNF (Нормальная форма Бойса-Кодда):
• Соответствует 3NF.
• Для каждой функциональной зависимости A → B атрибут A является потенциальным ключом.
• Устраняет аномалии при наличии перекрывающихся потенциальных ключей.
Нарушение BCNF: Расписание (Студент, Курс, Преподаватель), где Курс → Преподаватель, но Курс не является ключом. Ключ: (Студент, Курс).
Приведение к BCNF: разделить на (Студент, Курс) и (Курс, Преподаватель).

4NF (Четвертая нормальная форма):
• Соответствует BCNF.
• Отсутствуют нетривиальные многозначные зависимости (MVD).
• Если есть A →→ B, то A должен быть суперключом.
Нарушение 4NF: Преподаватель (Имя, Курс, Телефон) — один преподаватель ведет несколько курсов и имеет несколько телефонов независимо.
Приведение к 4NF: разделить на (Преподаватель, Курс) и (Преподаватель, Телефон).

5NF (Пятая нормальная форма / PJ/NF — Projection-Join Normal Form):
• Соответствует 4NF.
• Отсутствуют зависимости соединения — таблица не может быть декомпозирована на проекции без потерь информации.
• Каждая зависимость соединения определяется потенциальным ключом.
Применение: редко используется, актуально для сложных многосторонних связей.

6NF (Шестая нормальная форма):
• Максимальная декомпозиция — каждая таблица содержит первичный ключ и максимум один неключевой атрибут.
• Используется в темпоральных БД для хранения истории изменений.
• На практике применяется крайне редко из-за чрезмерной фрагментации.
Вопрос: Уровни (этапы) проектирования баз данных.
1. Инфологический (концептуальный): анализ предметной области, выявление сущностей, атрибутов и связей; создание ER-диаграмм; независим от СУБД.

2. Даталогический (логический): преобразование концептуальной модели в логическую схему (таблицы, ключи, связи); нормализация; учет особенностей типа СУБД (реляционная, NoSQL).

3. Физический: реализация в конкретной СУБД; создание индексов, партиций; оптимизация производительности; настройка хранения.
Вопрос: Оценка сложности проектирования базы данных. Выявление сущностей и атрибутов.
Оценка сложности проектирования:
Зависит от следующих факторов:
• Количество сущностей и атрибутов в предметной области.
• Количество и сложность связей между сущностями.
• Число и сложность бизнес-правил и ограничений целостности.
• Требования к производительности и масштабируемости.
• Объем данных и интенсивность операций.
• Количество пользователей и типов доступа.
• Требования к безопасности и аудиту.

Выявление сущностей:
• Анализ предметной области и бизнес-процессов.
• Определение объектов, о которых нужно хранить информацию (обычно существительные в описании).
• Проверка на уникальность — каждая сущность должна быть отличима от других.
• Проверка на значимость — сущность должна иметь бизнес-смысл.
• Отсечение слабых сущностей, которые могут быть атрибутами.
Пример: В системе библиотеки сущности: Книга, Читатель, Автор, Издательство, Выдача.

Выявление атрибутов:
• Определение характеристик каждой сущности (свойства, описывающие объект).
• Выбор первичного ключа — уникального идентификатора.
• Проверка атомарности — атрибут должен быть неделимым.
• Устранение вычисляемых атрибутов — не хранить то, что можно вычислить (возраст вместо даты рождения).
• Определение обязательности (NOT NULL) и значений по умолчанию.
Пример: Книга: ISBN (первичный ключ), Название, Год_издания, Количество_страниц, Издательство_ID (внешний ключ).
Вопрос: Использование инструментов проектирования баз данных на инфологическом уровне.
Инструменты: ERwin, MySQL Workbench, pgModeler, Lucidchart, Draw.io, Visual Paradigm.

Возможности: создание ER-диаграмм; определение сущностей, атрибутов, связей; визуализация кардинальности; документирование бизнес-правил; экспорт в даталогическую модель; совместная работа команды.
Вопрос: Проецирование инфологической модели на даталогический уровень. Использование инструментов проектирования баз данных на даталогическом уровне.
Проецирование: сущности → таблицы; атрибуты → столбцы; связи → внешние ключи; M:N → промежуточные таблицы; определение типов данных; применение нормализации.

Инструменты: MySQL Workbench, DBeaver, DataGrip, pgAdmin, SQL Server Management Studio.
Функции: создание схем БД; генерация DDL-скриптов; прямое и обратное проектирование; визуализация связей; управление индексами и ограничениями.
Вопрос: Оптимизация даталогической модели с учётом требований конкретных систем управления базами данных.
Оптимизация: выбор оптимальных типов данных для конкретной СУБД; денормализация для повышения производительности (контролируемая избыточность); использование специфичных возможностей СУБД (материализованные представления, партиционирование); учет ограничений СУБД (длина имен, типы индексов); настройка параметров хранения; планирование роста данных.
Вопрос: Принципы обеспечения эффективного функционирования баз данных на физическом уровне.
Принципы:
• Индексирование часто используемых столбцов (первичные, внешние ключи, поля поиска).
• Партиционирование больших таблиц (горизонтальное, вертикальное).
• Оптимизация запросов (анализ планов выполнения).
• Настройка буферов и кэша СУБД.
• Выбор типов хранения (InnoDB, MyISAM и др.).
• Регулярное обслуживание (VACUUM, ANALYZE, дефрагментация).
• Мониторинг производительности и узких мест.
Вопрос: Понятие языка структурированных запросов. Стандарты языка структурированных запросов.
SQL (Structured Query Language) — декларативный язык для управления реляционными БД: определение структур, манипулирование данными, управление доступом.

Стандарты: SQL-86 (первый стандарт ANSI), SQL-89, SQL-92 (SQL2), SQL:1999 (SQL3, добавлены триггеры, рекурсия), SQL:2003 (XML, оконные функции), SQL:2006, SQL:2008, SQL:2011 (темпоральные данные), SQL:2016 (JSON), SQL:2023 (последний).
Каждая СУБД реализует стандарт с расширениями.
Вопрос: Особенности реализации языка структурированных запросов в различных современных системах управления базами данных.
MySQL: диалект с расширениями (LIMIT, AUTO_INCREMENT); движки хранения (InnoDB, MyISAM).
PostgreSQL: строгое соответствие стандартам; расширенные типы (массивы, JSON, геоданные); мощные возможности (CTE, оконные функции).
Oracle: PL/SQL; продвинутые функции (аналитические, иерархические запросы); оптимизатор.
SQL Server: T-SQL; интеграция с .NET; специфичные функции (MERGE, OUTPUT).
SQLite: упрощенный SQL; ограниченные типы данных; отсутствие некоторых ALTER TABLE операций.
Вопрос: Операции управления структурами баз данных. Создание, модификация, удаление таблиц и представлений.
Операции с таблицами:
• CREATE TABLE — создание новой таблицы с определением столбцов, типов данных и ограничений.
• ALTER TABLE — модификация структуры: добавление/удаление столбцов, изменение типов данных, добавление/удаление ограничений.
• DROP TABLE — полное удаление таблицы и всех её данных.
• TRUNCATE TABLE — быстрая очистка всех данных с сохранением структуры таблицы.

Операции с представлениями (VIEW):
• CREATE VIEW — создание виртуальной таблицы на основе SELECT-запроса.
• ALTER VIEW — изменение определения существующего представления.
• DROP VIEW — удаление представления.

Примеры:
-- Создание таблицы
CREATE TABLE Employees (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(100) NOT NULL,
  salary DECIMAL(10,2),
  department_id INT,
  FOREIGN KEY (department_id) REFERENCES Departments(id)
);

-- Модификация таблицы
ALTER TABLE Employees ADD COLUMN email VARCHAR(100);
ALTER TABLE Employees MODIFY COLUMN salary DECIMAL(12,2);
ALTER TABLE Employees DROP COLUMN email;

-- Создание представления
CREATE VIEW HighEarners AS
SELECT name, salary FROM Employees WHERE salary > 100000;
Вопрос: Создание, модификация, удаление связей.
Создание связи:
• При создании таблицы: FOREIGN KEY (column) REFERENCES table(column).
• Для существующей таблицы: ALTER TABLE ADD CONSTRAINT имя FOREIGN KEY (column) REFERENCES table(column).

Модификация связи:
Прямой модификации нет — необходимо удалить старое ограничение и создать новое:
1. ALTER TABLE DROP CONSTRAINT имя_старого_ограничения.
2. ALTER TABLE ADD CONSTRAINT имя_нового_ограничения FOREIGN KEY ...

Удаление связи:
• MySQL: ALTER TABLE DROP FOREIGN KEY имя_ограничения.
• PostgreSQL/SQL Server: ALTER TABLE DROP CONSTRAINT имя_ограничения.

Опции ссылочной целостности:
• ON DELETE CASCADE — каскадное удаление связанных записей.
• ON DELETE SET NULL — установка NULL при удалении.
• ON DELETE RESTRICT — запрет удаления при наличии связанных записей.
• ON UPDATE CASCADE — каскадное обновление при изменении ключа.

Пример:
-- Создание связи при создании таблицы
CREATE TABLE Orders (
  id INT PRIMARY KEY,
  customer_id INT,
  FOREIGN KEY (customer_id) REFERENCES Customers(id)
    ON DELETE CASCADE ON UPDATE CASCADE
);

-- Добавление связи к существующей таблице
ALTER TABLE Orders
ADD CONSTRAINT fk_customer
FOREIGN KEY (customer_id) REFERENCES Customers(id)
ON DELETE SET NULL;
Вопрос: Операции выборки данных.
SELECT — основная операция выборки данных из БД.
Компоненты:
• SELECT — список столбцов или * (все столбцы).
• FROM — таблицы-источники данных.
• WHERE — условия фильтрации строк.
• JOIN — соединение таблиц (INNER, LEFT, RIGHT, FULL, CROSS).
• GROUP BY — группировка с агрегатными функциями (COUNT, SUM, AVG, MIN, MAX).
• HAVING — фильтрация групп после агрегации.
• ORDER BY — сортировка результатов (ASC, DESC).
• LIMIT/OFFSET — ограничение числа строк, пагинация.
• Подзапросы, UNION, INTERSECT, EXCEPT — комбинирование результатов.

Примеры:
-- Простая выборка с фильтрацией
SELECT name, salary FROM Employees WHERE salary > 50000;

-- JOIN двух таблиц
SELECT e.name, d.department_name
FROM Employees e
INNER JOIN Departments d ON e.department_id = d.id;

-- Группировка с агрегацией
SELECT department_id, COUNT(*) as emp_count, AVG(salary) as avg_salary
FROM Employees
GROUP BY department_id
HAVING COUNT(*) > 5;

-- Подзапрос
SELECT name FROM Employees
WHERE salary > (SELECT AVG(salary) FROM Employees);
Вопрос: Операции добавления, модификации, удаления данных.
INSERT: добавление новых строк в таблицу.
• INSERT INTO table (columns) VALUES (values) — вставка одной или нескольких строк.
• INSERT INTO table SELECT ... — вставка результата запроса.

UPDATE: изменение существующих данных.
• UPDATE table SET column = value WHERE условие — обновление строк, удовлетворяющих условию.

DELETE: удаление строк из таблицы.
• DELETE FROM table WHERE условие — удаление строк по условию.
• DELETE FROM table — удаление всех строк (медленнее TRUNCATE).

MERGE/UPSERT: вставка или обновление в зависимости от существования записи.
• MySQL: INSERT ... ON DUPLICATE KEY UPDATE
• PostgreSQL: INSERT ... ON CONFLICT DO UPDATE
• Oracle/SQL Server: MERGE INTO

Примеры:
-- INSERT
INSERT INTO Employees (name, salary, department_id)
VALUES ('Иван Петров', 60000, 3);

-- UPDATE
UPDATE Employees
SET salary = salary * 1.1
WHERE department_id = 5;

-- DELETE
DELETE FROM Employees
WHERE hire_date < '2020-01-01';

-- UPSERT (PostgreSQL)
INSERT INTO Products (id, name, price)
VALUES (1, 'Laptop', 1000)
ON CONFLICT (id) DO UPDATE SET price = EXCLUDED.price;
Вопрос: Использование представлений. Создание представлений для расширения возможностей реляционной базы данных.
Представление (VIEW) — виртуальная таблица, основанная на результате SQL-запроса. Не хранит данные физически, а выполняет запрос при обращении.

Преимущества:
• Упрощение сложных запросов — скрытие JOIN и подзапросов.
• Ограничение доступа к данным (безопасность) — показ только разрешенных столбцов/строк.
• Логическая независимость данных — изменение структуры таблиц без изменения приложений.
• Переиспользование логики — единожды определенный запрос используется многократно.

Создание: CREATE VIEW имя AS SELECT ...

Примеры:
-- Упрощение сложного запроса
CREATE VIEW EmployeeDetails AS
SELECT e.id, e.name, e.salary, d.department_name, d.location
FROM Employees e
JOIN Departments d ON e.department_id = d.id;

-- Использование представления
SELECT * FROM EmployeeDetails WHERE salary > 70000;

-- Ограничение доступа (только активные сотрудники)
CREATE VIEW ActiveEmployees AS
SELECT id, name, email FROM Employees WHERE status = 'active';
Обновляемые представления: позволяют INSERT/UPDATE/DELETE при соблюдении условий (одна базовая таблица, нет агрегации, DISTINCT, GROUP BY, UNION).

Материализованные представления (Materialized View): физически хранят результат запроса для ускорения доступа; требуют периодического обновления (REFRESH); поддерживаются в PostgreSQL, Oracle, SQL Server.
Вопрос: Использование хранимых подпрограмм. Создание хранимых функций для расширения возможностей реляционной базы данных. Создание хранимых процедур для расширения возможностей реляционной базы данных.
Хранимые подпрограммы — программный код (SQL и процедурные расширения), хранящийся и выполняющийся на сервере БД.

Хранимая функция (Stored Function):
• Возвращает одно значение определенного типа.
• Используется в SELECT, WHERE и других выражениях.
• Синтаксис: CREATE FUNCTION имя(параметры) RETURNS тип.
• Применение: вычисления, преобразования данных, инкапсуляция бизнес-логики.

Пример функции:
-- Функция расчета налога
CREATE FUNCTION calculate_tax(salary DECIMAL(10,2))
RETURNS DECIMAL(10,2)
DETERMINISTIC
BEGIN
  RETURN salary * 0.13;
END;

-- Использование
SELECT name, salary, calculate_tax(salary) as tax
FROM Employees;
Хранимая процедура (Stored Procedure):
• Выполняет действия, может возвращать несколько результатов или не возвращать ничего.
• Вызывается через CALL или EXECUTE.
• Синтаксис: CREATE PROCEDURE имя(параметры).
• Применение: сложная бизнес-логика, пакетные операции, управление транзакциями, модификация данных.

Пример процедуры:
-- Процедура повышения зарплаты
CREATE PROCEDURE raise_salary(IN dept_id INT, IN percent DECIMAL(5,2))
BEGIN
  UPDATE Employees
  SET salary = salary * (1 + percent / 100)
  WHERE department_id = dept_id;
END;

-- Вызов
CALL raise_salary(5, 10); -- повысить зарплату в отделе 5 на 10%
Преимущества:
• Производительность — компиляция и кэширование на сервере, снижение сетевого трафика.
• Безопасность — контроль доступа через права на выполнение, скрытие логики.
• Централизация логики — единая точка изменения бизнес-правил.
• Повторное использование кода — вызов из разных приложений.
• Атомарность операций — выполнение в рамках транзакции.
Вопрос: Создание триггеров для расширения возможностей реляционной базы данных.
Триггер (Trigger) — хранимая процедура, автоматически выполняющаяся при наступлении определенного события (INSERT, UPDATE, DELETE) на таблице.

Синтаксис создания: CREATE TRIGGER имя BEFORE/AFTER событие ON таблица FOR EACH ROW BEGIN ... END.

Типы по времени выполнения:
• BEFORE — выполняется до операции, позволяет изменить или отменить данные перед записью.
• AFTER — выполняется после операции, используется для аудита и каскадных действий.

Уровни срабатывания:
• FOR EACH ROW (строчный) — срабатывает для каждой затронутой строки.
• FOR EACH STATEMENT (операторный) — срабатывает один раз на всю операцию.

Применение:
• Автоматическое заполнение полей (даты создания/изменения, версии, пользователь).
• Проверка сложных ограничений целостности, не выразимых через CHECK.
• Аудит изменений — логирование операций в отдельную таблицу.
• Каскадные операции — автоматическое обновление связанных данных.
• Синхронизация данных между таблицами.
• Ведение истории изменений (версионирование).

Примеры:
-- Автоматическое заполнение даты изменения (MySQL)
CREATE TRIGGER before_employee_update
BEFORE UPDATE ON Employees
FOR EACH ROW
BEGIN
  SET NEW.updated_at = NOW();
END;

-- Аудит удалений (PostgreSQL)
CREATE TRIGGER audit_employee_delete
AFTER DELETE ON Employees
FOR EACH ROW
EXECUTE FUNCTION log_deletion();

CREATE FUNCTION log_deletion() RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO Audit_Log (table_name, record_id, action, deleted_at)
  VALUES ('Employees', OLD.id, 'DELETE', NOW());
  RETURN OLD;
END;
$$ LANGUAGE plpgsql;

-- Проверка бизнес-правила
CREATE TRIGGER check_salary_limit
BEFORE INSERT ON Employees
FOR EACH ROW
BEGIN
  IF NEW.salary > 200000 THEN
    SIGNAL SQLSTATE '45000'
    SET MESSAGE_TEXT = 'Зарплата превышает лимит';
  END IF;
END;
Важные замечания:
• Могут снизить производительность при частых операциях.
• Усложняют отладку — скрытая логика.
• Избегать рекурсивных триггеров (триггер вызывает сам себя).
• Документировать все триггеры для понимания поведения БД.
• NEW — новые значения (INSERT, UPDATE), OLD — старые значения (UPDATE, DELETE).
